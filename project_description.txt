Техническое задание
на разработку системы оптимизации подсортировки товарных остатков и пополнения розничной сети
Заказчик: Mavi Jeans
Предмет автоматизации: распределение товарных остатков между центральным складом и розничными магазинами на основе данных из 1С:Управление торговлей
1. Общая информация о проекте
1.1. Наименование проекта
Разработка системы автоматизированной оптимизации подсортировки товарных остатков и пополнения розничной сети.
1.2. Цель проекта
Создать систему, которая автоматически рассчитывает оптимальный план распределения товарных остатков между центральным складом и магазинами розничной сети с учетом истории продаж, сезонности, коллекций, партий, ограничений магазинов и различий в локальном спросе.
1.3. Основание для разработки
Необходимость повышения точности пополнения магазинов, снижения пересортицы и затоваривания, а также перевода процесса распределения товара из ручного или полуэкспертного режима в системный, математически обоснованный контур планирования.
2. Бизнес-контекст и проблема
На момент старта проекта у заказчика существовали следующие проблемы:
    1. Отсутствовал единый механизм распределения товара между центральным складом и магазинами.
    2. Решения о пополнении часто принимались вручную либо по упрощенным правилам.
    3. Не учитывался в полном объеме весь набор ограничений:
        ◦ сезонность;
        ◦ коллекции;
        ◦ партии;
        ◦ остатки;
        ◦ вместимость торговых точек;
        ◦ различия в спросе по регионам и локациям;
        ◦ приоритетность магазинов.
    4. В части магазинов наблюдался дефицит нужных SKU, в других — накопление избыточных остатков.
    5. Ошибки распределения приводили к:
        ◦ упущенным продажам;
        ◦ снижению доступности нужного ассортимента;
        ◦ замораживанию оборотного капитала в избыточных остатках;
        ◦ росту markdown / уценки по сезонным коллекциям.
3. Цели автоматизации
Система должна была обеспечить:
    • автоматический расчет потребности магазинов в товаре;
    • автоматическое формирование плана пополнения и перемещений;
    • учет ограничений по партиям, коллекциям, остаткам и вместимости;
    • учет сезонных паттернов спроса;
    • учет различий в востребованности ассортимента по локациям;
    • снижение пересортицы;
    • снижение затоваривания;
    • повышение доступности нужного ассортимента в магазинах;
    • поддержку операционного исполнения через выгрузку результатов в пригодном для работы виде.
4. Объект автоматизации
Объектом автоматизации является процесс планирования подсортировки и пополнения товарных остатков розничной сети одежды, включающий:
    • анализ остатков на центральном складе;
    • анализ остатков в магазинах;
    • анализ истории продаж;
    • прогнозирование потребности по магазинам и SKU;
    • распределение доступного товара между магазинами;
    • формирование исполнимого плана отгрузок и перемещений.

5. Источники данных
Основным источником данных выступает 1С:Управление торговлей.
5.1. Система должна использовать следующие данные:
    1. История продаж по магазинам, товарам, периодам.
    2. Текущие остатки на центральном складе.
    3. Текущие остатки по магазинам.
    4. Данные по SKU:
        ◦ артикул;
        ◦ категория;
        ◦ коллекция;
        ◦ сезон;
        ◦ принадлежность к партии;
        ◦ доступность к распределению.
    5. Целевые уровни запасов.
    6. Показатели оборачиваемости.
    7. Ограничения по вместимости торговых точек.
    8. Приоритетность магазинов.
    9. Ассортиментные правила.
    10. Признаки локальной востребованности ассортимента.
5.2. Требования к качеству входных данных
Входные данные должны быть:
    • полными в рамках расчетного периода;
    • непротиворечивыми;
    • синхронизированными по ключевым справочникам;
    • пригодными для автоматизированного расчета без ручной очистки в момент запуска.
6. Функциональные требования к системе
6.1. Загрузка и подготовка данных
Система должна:
    • получать данные из 1С:Управление торговлей;
    • выполнять валидацию входных данных;
    • проверять полноту и корректность обязательных полей;
    • формировать единый расчетный контур по товарам, магазинам и остаткам.
6.2. Анализ потребности магазинов
Система должна рассчитывать потребность магазинов в пополнении с учетом:
    • истории продаж;
    • сезонности;
    • текущих остатков;
    • целевых уровней запасов;
    • локальной динамики спроса;
    • специфики конкретной точки продаж.
6.3. Прогнозирование спроса
Система должна использовать прогнозную модель, позволяющую:
    • оценивать ожидаемую потребность по SKU в каждом магазине;
    • учитывать сезонные паттерны;
    • учитывать региональные и локационные особенности спроса;
    • выделять товары с высокой вероятностью востребованности.
6.4. Классификация товаров и магазинов
Система должна выполнять классификацию:
    • товаров — по степени приоритетности к распределению;
    • магазинов — по приоритету обеспечения ассортиментом;
    • сочетаний “товар-магазин” — по вероятности востребованности и целесообразности пополнения.
6.5. Оптимизационный расчет
Система должна автоматически рассчитывать оптимальный план распределения товара с учетом:
    • доступных остатков;
    • ограничений по партиям;
    • ограничений по коллекциям;
    • сезонности;
    • вместимости магазинов;
    • приоритетности магазинов;
    • целевых уровней запасов;
    • необходимости снижения пересортицы;
    • необходимости снижения затоваривания.
6.6. Формирование плана пополнения
На выходе система должна формировать:
    • перечень товаров к отгрузке;
    • магазин-получатель;
    • рекомендуемое количество;
    • основание распределения;
    • приоритет исполнения;
    • признак источника пополнения;
    • перечень внутренних перемещений, если они предусмотрены логикой процесса.
6.7. Подготовка результатов к операционному исполнению
Результаты расчета должны быть пригодны для:
    • резервирования товара на центральном складе;
    • формирования документов на отгрузку;
    • формирования документов внутренних перемещений;
    • передачи в операционную логистику и коммерческий блок.
7. Логика работы системы
7.1. Общая схема
    1. Загрузка данных из 1С.
    2. Проверка корректности и полноты данных.
    3. Расчет потребности магазинов.
    4. Прогнозирование спроса по SKU и магазинам.
    5. Классификация товаров и магазинов по приоритетам.
    6. Применение ограничений и бизнес-правил.
    7. Оптимизационный расчет плана распределения.
    8. Формирование итогового плана пополнения и перемещений.
    9. Выгрузка результата в пригодном для исполнения формате.
7.2. Цели оптимизации
При расчете система должна балансировать несколько целей одновременно:
    • максимизация доступности товара в магазинах;
    • снижение избыточных запасов;
    • соблюдение ограничений по коллекциям и партиям;
    • повышение точности пополнения;
    • снижение вероятности недопоставки востребованных SKU;
    • снижение риска накопления неликвидных остатков.
7.3. Класс решаемой задачи
По сути система решает задачу, близкую к классу production planning / scheduling, где есть:
    • ограниченные ресурсы;
    • набор ограничений;
    • приоритеты;
    • правила распределения;
    • множество точек назначения;
    • необходимость построить наилучший исполнимый план.
8. Бизнес-правила и ограничения
Система должна учитывать следующие ограничения:
8.1. Ограничения по товару
    • товар может принадлежать определенной партии;
    • товар может относиться к конкретной коллекции;
    • товар может быть сезонно ограничен;
    • часть товара может быть недоступна к распределению;
    • для отдельных категорий товара могут действовать дополнительные ассортиментные ограничения.
8.2. Ограничения по магазинам
    • ограниченная вместимость торговой точки;
    • разная приоритетность магазинов;
    • разный профиль спроса;
    • разная скорость оборачиваемости;
    • различия в целевом ассортименте.
8.3. Ограничения по распределению
    • нельзя распределить больше доступного остатка;
    • нельзя нарушать ограничения по коллекциям и партиям;
    • нельзя формировать поставку, ведущую к экономически необоснованному затовариванию;
    • при дефиците товара распределение должно происходить по заданным приоритетам.
9. Выходные данные системы
9.1. Состав результата
Система должна формировать итоговый план, содержащий:
    • магазин;
    • SKU;
    • рекомендуемое количество к пополнению;
    • источник пополнения;
    • приоритет;
    • расчетное обоснование;
    • ограничения, повлиявшие на решение;
    • итоговый статус рекомендации.
9.2. Формат представления
Результат должен быть доступен в виде:
    • таблицы расчета;
    • выгрузки для последующей загрузки или обработки в учетной системе;
    • набора данных, пригодного для формирования документов на отгрузку и перемещение.
10. Требования к интеграции
10.1. Интеграция с 1С:Управление торговлей
Система должна обеспечивать:
    • получение данных из 1С;
    • сопоставление справочников;
    • использование актуальных остатков и истории продаж;
    • выгрузку результатов расчета в формате, пригодном для дальнейшего использования в операционном контуре.
10.2. Обмен данными
Должны быть предусмотрены:
    • регламент загрузки данных;
    • формат выгрузки результатов;
    • правила обработки ошибок и неполных данных;
    • журнал выполнения расчета.
11. Нефункциональные требования
11.1. Надежность
Система должна выполнять расчет стабильно при наличии корректных входных данных.
11.2. Масштабируемость
Система должна поддерживать работу с ассортиментом и количеством магазинов, характерными для федеральной или региональной розничной сети fashion-сегмента.
11.3. Повторяемость результата
При одинаковом наборе входных данных и идентичных параметрах расчета система должна формировать воспроизводимый результат.
11.4. Прозрачность
Результаты должны быть интерпретируемыми для бизнеса: по каждой рекомендации должна быть возможность понять, почему именно такое распределение было предложено.
11.5. Удобство операционного использования
Итог расчета должен быть пригоден для практического использования без существенной ручной доработки.
12. Пользователи системы
Основными пользователями решения являются:
    • отдел коммерческого планирования;
    • отдел управления ассортиментом;
    • логистика / складская функция;
    • сотрудники, отвечающие за пополнение магазинов;
    • аналитики и операционные менеджеры.
13. Ожидаемый эффект от внедрения
В результате внедрения заказчик должен получить инструмент, позволяющий:
    • повысить доступность нужного ассортимента в магазинах;
    • учитывать специфику сезонных коллекций и партий;
    • снижать пересортицу;
    • уменьшать затоваривание;
    • принимать решения о пополнении на основе данных и прогноза, а не только экспертных правил;
    • повысить качество распределения товара между точками продаж;
    • улучшить оборачиваемость запасов;
    • снизить объем markdown / уценки;
    • поддержать выручку в рамках сезонных продаж.
14. Критерии приемки результата
Результат проекта считается приемлемым, если:
    1. Система использует данные из 1С:Управление торговлей.
    2. Система автоматически формирует план пополнения магазинов.
    3. В расчете учитываются:
        ◦ остатки;
        ◦ история продаж;
        ◦ сезонность;
        ◦ коллекции;
        ◦ партии;
        ◦ ограничения магазинов;
        ◦ приоритетность магазинов.
    4. Итоговый план пригоден для операционного исполнения.
    5. Результат позволяет сократить объем ручной корректировки.
    6. Система поддерживает принятие решений по распределению товара на основе набора бизнес-ограничений и прогноза.
15. Практическая ценность проекта
Проект переводит процесс распределения товара из ручного или полуэкспертного режима в единый системный контур планирования. Для fashion-ритейла это особенно важно, так как качество распределения напрямую влияет на:
    • сезонную выручку;
    • оборачиваемость запасов;
    • уровень уценки;
    • полноту и корректность представленности коллекции в конкретной локации;
    • эффективность коммерческого и логистического управления.
16. Резюме проекта
В рамках проекта для Mavi Jeans была реализована система, которая на основании данных из 1С:Управление торговлей автоматически рассчитывала, какой товар, в каком количестве и в какие магазины необходимо направить. Решение учитывало продажи, остатки, сезонность, партии, коллекции, вместимость торговых точек, приоритетность магазинов и особенности локального спроса.
На выходе заказчик получал готовый к использованию план пополнения и перемещений, который можно было использовать в операционной работе практически без ручной доработки.
